# TRM評価軸（汎用版）

> **目的**: sakura-editor実証で用いた評価軸を抽象化し、任意のリポジトリに適用可能な形にする
> **背景**: 論文v3フレーミング「品質情報の可読性ギャップ」に対応。TRMの有効性を**網羅性**と**可読性**の2軸で測る
> **関連**: `../experiments/evaluation-report.md`（sakura-editor実績）、`../experiments/additional-analysis.md`（可読性分析）

---

## 1. 評価軸の全体像

TRMの品質は以下の4カテゴリで測定する:

| カテゴリ | 観点 | 代表指標 | 目的 |
|---|---|---|---|
| A. 網羅性 | 対象コードに対する要求密度 | カバー率、要求密度 | テスト漏れの定量化 |
| B. 可読性 | 非開発者が理解できる度合 | 可読率（Level 1+2 の割合） | 品質情報のコミュニケーション基盤性 |
| C. 検証性 | テストとして成立する度合 | 初回PASS率、期待値根拠トレース率 | 要求が実装検証として機能するか |
| D. 保守性 | 変更追随性・再利用性 | トレーサビリティ密度、影響範囲推定可能性 | 仕様変更に対するコストの低さ |

### 適用順序

1. 最低限: A（網羅性）+ B（可読性）
2. 論文・社内導入評価: A + B + C
3. 継続運用評価: A + B + C + D

## 2. カテゴリA: 網羅性の評価軸

### A-1. 要求数

| 指標 | 定義 | 集計単位 | 汎用性 |
|---|---|---|---|
| 総要求数 | TRM YAML内の `test_requirements[]` の総数 | TRM全体 | 任意リポジトリで算出可 |
| 関数あたり要求数 | 総要求数 / 選定関数数 | 関数選定結果 | 同上 |
| 行あたり要求密度 | 総要求数 / 対象関数の合計行数 | コード行数 | 同上 |
| 種別内訳 | BR/EC/BV/ER/DP/CI/SV/CP 各件数 | `type` フィールド | 同上 |

### A-2. カバー率

| 指標 | 計算式 | 補足 |
|---|---|---|
| 要求カバー率 | カバー済み要求数 ÷ 総要求数 × 100% | `traceability-matrix.md` の対応件数から算出 |
| 種別別カバー率 | 各種別ごとに同上 | 種別ごとに偏りが出ていないか確認 |
| 優先度別カバー率 | 各priority（high/medium/low）ごとに同上 | highが最優先で100%を目指す |

### A-3. 既存テストとの比較（任意）

| 指標 | 定義 |
|---|---|
| 既存カバー重複率 | TRMで導出した要求のうち、既存テストにも対応がある件数 / 総要求数 |
| TRM固有カバー率 | 既存テストに対応がなく、TRMで初めて追加された要求 / 総要求数 |
| 見逃し検出数 | TRMで発見したが既存テストに存在しない観点の件数 |

### A-4. OOP拡張の追加指標（v3.0/v3.1、oop_analysis 有効時）

| 指標 | 定義 | 目的 |
|---|---|---|
| CI 網羅率 | CI要求数 / 対象クラスのオーバーライドメソッド総数 | 多態性検証の充足度 |
| SV 網羅率 | SV要求数 / 対象クラスの可変フィールド総数 | 状態検証の充足度 |
| CP 検出率 | CP要求数 / 検出パターン数 | パターン使用箇所の検証充足度 |
| 階層深度カバー | テスト対象が触る最大継承深度 | 浅い継承に偏っていないか |
| **EN 網羅率** (v3.1) | EN要求数 / 対象クラスのfield総数 | メンバ構造設計の検証充足度 |
| **アクセス制御検証率** (v3.1) | access_control_correctness 要求数 / 非public期待field数 | カプセル化の設計整合性 |
| **構築契約カバー** (v3.1) | construction_contract 要求数 / コンストラクタ × 必須field の組合せ数 | 初期化契約の完全性 |
| **カプセル化リスク消化率** (v3.1) | 対応要求のあるリスク / 検出されたリスク総数 | 検出された設計不備への対応度 |

**v3.1 追加の SV サブタイプ指標:**
- `member_declaration_validity`: 宣言妥当性が検証された field / 総 field
- `member_initialization_requirement`: 初期化必須性が検証された field / `required_at_construction: true` の field 総数

## 3. カテゴリB: 可読性の評価軸

### B-1. 3レベル分類（sakura-editor実証で採用）

| レベル | 判定基準 | 意味 |
|---|---|---|
| Level 1 | 一般用語のみで理解可能 | コード知識不要 |
| Level 2 | ドメイン用語が含まれるが、その分野の人なら理解可能 | ドメイン知識のみ必要 |
| Level 3 | 変数名・型名・制御構造等の実装語彙を含む | コード知識が必要 |

### B-2. 判定方法

要求の `description` フィールドを対象に、以下のキーワード混入で判定:

| レベル判定 | 検出ルール例 |
|---|---|
| Level 3候補 | `NULL`, `break`, `else`, `goto`, `offset`, 型名（`int`, `size_t`, 列挙型名）、コード中の変数名 |
| Level 2候補 | ドメイン用語（例: 「メールアドレス」「バージョン」「エスケープ」）、テスト工学用語（例: 「境界値」「0パディング」） |
| Level 1候補 | 上記いずれも含まない平易な記述 |

**実装方針**: 自動判定スクリプトを準備し、3段階のうちどれに該当するかをLLMに判定させる + 人手レビューで補正。

### B-3. 集計指標

| 指標 | 計算式 | 目的 |
|---|---|---|
| 可読率（L1+L2） | (Level 1 件数 + Level 2 件数) / 総件数 × 100% | 非開発者への伝達性 |
| コード知識純率（L3） | Level 3 件数 / 総件数 × 100% | 専門性の度合 |
| 種別別可読率 | BR/EC/BV/ER/DP/CI/SV/CP 各種別ごとに L1+L2 割合 | 種別による可読性の差異把握 |
| 領域別可読率 | ファイル/モジュール単位で L1+L2 割合 | ドメイン特性の差異把握 |

### B-4. 目標値（参考）

| 状況 | 可読率目標 |
|---|---|
| 品質情報を非開発者と共有する基盤として成立 | ≥ 60% |
| sakura-editor実績（論文値） | 65.7% |
| 成熟した設計・ドメイン主導のリポジトリ | ≥ 75% |

## 4. カテゴリC: 検証性の評価軸

### C-1. 初回PASS率

| 指標 | 定義 |
|---|---|
| 初回PASS率 | 生成テストを修正なしで実行し成功した件数 / 実行件数 |
| 修正後PASS率 | 期待値等を調整後の成功率（通常100%が目標） |
| 修正の種類内訳 | 期待値修正 / スタブ修正 / シグネチャ修正 / 入力修正 の件数 |

### C-2. 期待値根拠のトレース率

| 指標 | 定義 |
|---|---|
| 根拠ありテスト率 | 期待値がソース実装から導出されているテスト / 総テスト数 |
| LLM推論依存率 | 期待値がLLM推論のみに依存しているテスト / 総テスト数 |

**基準**: LLM推論依存率はゼロを目指す（CLAUDE.md記載の制約）。

### C-3. トレーサビリティ整合性

| 指標 | 定義 |
|---|---|
| TRM→テスト対応率 | `traceability-matrix.md` でテストにマップ済みの要求数 / 総要求数 |
| 孤立テスト率 | TRMに対応する要求がないテスト / 総テスト数 |

---

## 5. カテゴリD: 保守性の評価軸

### D-1. 変更影響推定

| 指標 | 定義 | 算出方法 |
|---|---|---|
| 関数変更影響推定 | ある関数が変わった時に影響を受ける要求の数 | `source_ref` の逆引き |
| 要求変更影響推定 | ある要求が変わった時に影響を受けるテストの数 | `traceability-matrix.md` の逆引き |

### D-2. 再利用性

| 指標 | 定義 |
|---|---|
| 汎用要求比率 | プロジェクト非依存の要求（境界値・エラーパス等）/ 総要求数 |
| ドメイン固有要求比率 | プロジェクト固有知識を含む要求 / 総要求数 |

## 6. 評価フローテンプレート

任意のリポジトリにパイプラインを適用した際の評価レポート構成。

```
evaluation-report.md
├── 1. 実験条件
│   ├── 1.1 対象リポジトリ（URL・言語・FW・規模）
│   ├── 1.2 対象関数・クラス一覧
│   └── 1.3 選定基準
├── 2. カテゴリA: 網羅性
│   ├── 2.1 要求数（A-1）
│   ├── 2.2 カバー率（A-2）
│   ├── 2.3 既存テスト比較（A-3、任意）
│   └── 2.4 OOP指標（A-4、oop_analysis有効時）
├── 3. カテゴリB: 可読性
│   ├── 3.1 3レベル分類結果
│   ├── 3.2 全体可読率
│   ├── 3.3 種別別・領域別の可読率
│   └── 3.4 目標値との比較
├── 4. カテゴリC: 検証性
│   ├── 4.1 初回PASS率
│   ├── 4.2 期待値根拠トレース
│   └── 4.3 トレーサビリティ整合性
├── 5. カテゴリD: 保守性（任意）
└── 6. 考察・結論
```

## 7. 評価軸チェックリスト（新規リポジトリ適用時）

### 7.1 最低限（A+B）

- [ ] `test-requirements.yaml` の総要求数を集計
- [ ] 種別内訳（BR/EC/BV/ER/DP、OOP有効なら+CI/SV/CP）を集計
- [ ] `traceability-matrix.md` からカバー率を算出
- [ ] 各要求の `description` に可読性レベル（1/2/3）をタグ付け
- [ ] 可読率（L1+L2）の全体値・種別別値・領域別値を算出

### 7.2 論文・社内導入評価（A+B+C）

- [ ] 生成テストを実行し、初回PASS率を測定
- [ ] 失敗テストを修正し、修正種類を分類
- [ ] 期待値がLLM推論のみに依存するケースを検出
- [ ] TRM→テストのトレーサビリティ整合性を確認

### 7.3 継続運用評価（A+B+C+D）

- [ ] 関数変更時の要求影響範囲推定
- [ ] 要求変更時のテスト影響範囲推定
- [ ] 汎用要求 vs ドメイン固有要求の比率を算出

## 8. 言語・ドメイン横断の正規化上の注意

| 項目 | 問題 | 対応 |
|---|---|---|
| 言語差異 | C++はマクロ展開でBR数が水増し | マクロ展開前と後の両方でカウントし注記 |
| ドメイン用語の強弱 | 金融/医療等は専門用語が多く Level 1 が減る | 目標可読率をドメイン別に調整（例: 金融は 50% 以上） |
| OOP vs 手続き型 | 手続き型は CI/SV/CP が 0 件になる | `oop_analysis.enabled: false` で従来5種別だけで評価 |
| 関数規模の不均衡 | 1関数300行vs30行で要求密度が大きく変動 | 関数規模別に層別集計し、単純平均を避ける |
