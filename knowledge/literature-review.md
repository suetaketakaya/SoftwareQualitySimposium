# 先行研究調査: LLM活用開発における品質保証の可視化

**調査日**: 2026年3月31日
**対象論文期間**: 2023年〜2026年

---

## 1. 課題の学術的位置づけ

本研究が扱う課題「LLM活用開発における品質保証情報を、コード非専門者を含むステークホルダーに対して可視化・構造化する手法」は、以下の複数の研究分野の交差点に位置する。

### 1.1 所属する研究分野

| 分野 | 本研究との関係 |
|------|---------------|
| **LLM-based Software Engineering** | LLMによるコード生成・テスト生成の品質課題を前提として扱う |
| **Software Quality Assurance** | 単体テスト・コンポーネントテストレベルでの品質保証を目的とする |
| **Requirements-to-Test Traceability** | 要求からテストへの追跡可能性を構造化中間成果物で実現する |
| **Test Design Visualization** | テスト設計情報をコード非専門者に理解可能な形で可視化する |
| **Model-Based Testing (MBT)** | テスト要求モデル（YAML形式）をテスト設計のモデルとして導入する |
| **Specification by Example / BDD** | ステークホルダーが可読なテスト仕様という思想を共有する |

### 1.2 既存研究の空白領域

既存のLLMテスト生成研究は「**テスト生成の精度向上**」（カバレッジ、コンパイル成功率、バグ検出率）に集中している。一方、本研究が焦点とする「**生成されたテストの品質情報を非開発者が理解・レビュー可能にする**」という課題は、体系的に取り組まれた先行研究がほぼ存在しない。

具体的には:
- LLMテスト生成研究は「テストコードの品質」を測定するが、「テストの意図・網羅性を人間が理解できるか」は評価していない
- Requirements Traceability研究は要求-テスト間の追跡を扱うが、LLM生成コードに対する適用は未開拓
- BDD/Specification by Exampleは可読性の思想を持つが、単体テスト・コンポーネントテストレベルの構造的品質情報の可視化は対象外
- 品質ダッシュボード研究はメトリクスの可視化を扱うが、個々のテスト要求レベルの意味的な可読性は扱っていない

## 3. 比較対象手法の詳細

### 3.1 LLMテスト生成手法の比較

| 手法 | 発表 | アプローチ | カバレッジ | バグ検出 | 可読性/可視化 |
|------|------|-----------|-----------|---------|-------------|
| **CodaMOSA** | ICSE 2023 | SBST + LLM ハイブリッド。カバレッジ停滞時にLLM（Codex）にテスト例を要求 | 486ベンチマークで統計的に有意な改善（173/279ベンチマーク） | SBSTベースラインより改善 | **扱わない** |
| **ChatUniTest** | FSE 2024 | 適応的focal context機構 + 生成-検証-修復サイクル | ライン被覆率59.6%（EvoSuite, TestSpark超過） | EvoSuiteと同等以上 | **扱わない** |
| **TestPilot** | IEEE TSE 2024 | ゼロショットLLMプロンプト + 失敗テストの再プロンプト。追加学習不要 | 文カバレッジ中央値70.2%、分岐52.8% | Nessieより改善 | **扱わない** |
| **CoverUp** | 2024 | カバレッジ解析 + LLM反復会話。カバレッジ情報をプロンプトに統合 | モジュール中央値80%（CodaMOSA 47%を大幅超過） | 限定的 | **扱わない** |
| **CoverAgent (Qodo)** | 2024 | カバレッジ目標設定 + 反復生成。TestRunner/CoverageParser/PromptBuilder/AICallerの4コンポーネント | 目標カバレッジまで反復 | バグ発見よりカバレッジ向上を優先する設計上の限界あり | **扱わない** |
| **TELPA** | 2024 | プログラム解析でhard-to-coverブランチに特化。反復的反例ベースの改善 | 分岐カバレッジ平均31.39%改善（SBST比） | 改善あり | **扱わない** |
| **HardTestGen** | ICLR 2026 | 高品質テスト入出力の生成手法。RL訓練との組み合わせ | -- | 微妙なバグの検出に特化 | **扱わない** |
| **Meta ACH** | FSE 2025 | 変異誘導型LLMテスト生成。Fault Generator/Equivalence Detector/Test Generatorの3エージェント | 10,795クラスに適用、571テスト生成 | エンジニアが73%を採用、36%がプライバシー関連として妥当と判断 | **限定的**（エンジニア向け） |
| **LLMLOOP** | ICSME 2025 | 5つの反復フィードバックループ（コンパイル/静的解析/テスト失敗/突然変異解析） | pass@1: 80.85%, pass@10: 90.24% | 突然変異解析ベースで評価 | **扱わない** |
| **本研究（テスト要求モデル）** | SQiP 2026 | テスト要求モデル（YAML）を中間成果物として導入。静的解析+LLMで品質情報を構造化 | テスト生成のカバレッジも測定 | テスト要求の網羅性で評価 | **主要な貢献** |

### 3.2 比較の要点

上記すべての手法は「**テスト生成の自動化・精度向上**」を主眼としており、「可読性/可視化」列がすべて「扱わない」である点が重要である。本研究の独自性は、テスト生成の品質だけでなく、**その結果を非開発者が理解できる形で構造化・可視化する**点にある。

### 3.3 手法分類による位置づけ

Chu et al. (2025)のサーベイによる分類に基づくと:

```
テスト生成フェーズ:
├── 生成段階 (Generative Phase)
│   ├── プロンプトエンジニアリング (89%)  ← CodaMOSA, ChatUniTest, TestPilot等
│   ├── ファインチューニング
│   └── 強化学習  ← HardTestGen
│
├── 品質保証段階 (QA Phase)
│   ├── 反復的検証・修復ループ  ← CoverUp, CoverAgent, LLMLOOP
│   ├── 静的解析フィードバック  ← TELPA, LLMLOOP
│   └── 突然変異解析  ← Meta ACH, LLMLOOP
│
└── 【本研究の位置づけ: 新たな段階】
    └── 品質情報可視化段階 (Quality Information Visualization Phase)
        ├── テスト要求モデルによる構造化
        ├── 要求-テスト間トレーサビリティの可視化
        └── 非開発者向けレビュー可能性の担保
```

## 4. 本研究の独自性

### 4.1 先行研究との差異の明確化

| 差異の観点 | 先行研究の立場 | 本研究の立場 |
|-----------|--------------|-------------|
| **主要関心事** | テスト生成の精度向上（カバレッジ、コンパイル率） | 品質情報の可読性・可視化 |
| **対象読者** | 開発者・テストエンジニア | 要求作成ロール（非開発者含む） |
| **中間成果物** | なし（直接テストコードを生成） | テスト要求モデル（YAML形式） |
| **品質の表現** | コードレベル（カバレッジ%、Pass/Fail） | 要求レベル（「XX件のテスト要求のうちYY件がカバー」） |
| **トレーサビリティ** | 暗黙的（コード内の対応関係） | 明示的（要求→テスト要求→テストコードの追跡） |
| **レビュー可能性** | テストコードを読む必要あり | YAMLテスト要求をレビュー可能 |
| **テストオラクル問題** | 「実装された動作」を捉えがち | 「期待される動作」をテスト要求として明示的に記述 |
| **テストスメル対策** | 事後的にリファクタリング | テスト要求の段階で意図を構造化し、スメルの原因を予防 |

### 4.2 独自性の主張（3つの柱）

**柱1: 品質情報の可読性ギャップへの対処**

LLMテスト生成研究は「テストの品質」を測定するが、「テストの意図・網羅性を人間（特に非開発者）が理解できるか」という問題には取り組んでいない。本研究はテスト要求モデルを導入し、コード非専門者でも「何がテストされているか」「どの分岐がカバーされているか」をレビュー可能にする。

**柱2: LLM活用開発特有の品質保証プロセスの提案**

従来の品質保証は、開発者がコードを読んで品質情報を把握することを暗黙の前提としていた。LLM活用開発ではこの前提が崩れる。本研究は、LLMが生成したコードの品質情報を、コードを読まずに把握するための構造化手法を初めて提案する。

**柱3: テストオラクル問題の構造的緩和**

先行研究（2024）は、LLM生成テストのアサーションが「実装された動作」を捉えがちで「期待される動作」との乖離があることを指摘している。本研究のテスト要求モデルは、テスト生成に先立って「期待される動作」を明示的に記述するため、テストオラクル問題を構造的に緩和する。

### 4.3 BDD/Specification by Exampleとの差異

BDD（振る舞い駆動開発）やSpecification by Exampleは「非技術者にも可読なテスト仕様」という思想を共有するが、以下の点で本研究と異なる:

| 観点 | BDD/SbE | 本研究 |
|------|---------|--------|
| テストレベル | 受入テスト・システムテスト | 単体テスト・コンポーネントテスト |
| 記述対象 | ビジネス要件の具体例 | コード構造に基づく品質情報（分岐、境界値、エラー処理等） |
| 情報源 | ステークホルダーとの協議 | 静的解析+LLMによるコード構造解析 |
| 自動化の方向 | 仕様→テスト | コード→テスト要求→テスト |
| LLMとの関係 | LLM活用を前提としない | LLM活用開発のワークフローに組み込み |

### 4.4 ACL 2025 "Multi-Step Generation of Test Specifications using Large Language Models" との差異

ACL 2025のindustry trackで発表された "Multi-Step Generation of Test Specifications using Large Language Models" は、自然言語要求からテスト仕様を多段階で生成する手法を提案しており、本研究と最も近い先行研究である。ただし以下の重要な差異がある:

| 観点 | ACL 2025論文 | 本研究 |
|------|-------------|--------|
| 入力 | 自然言語要求仕様 | 既存コードベース |
| 方向性 | 要求→テスト仕様（トップダウン） | コード→テスト要求モデル→テスト（ボトムアップ+構造化） |
| 対象 | 機能安全要求のテスト仕様 | 単体テスト・コンポーネントテストの品質情報 |
| 品質情報 | テスト目的の生成 | カバレッジ、静的解析結果、処理フローを含む包括的品質情報 |
| 可読性の目的 | テスト仕様の品質向上 | コード非専門者によるレビュー可能性の実現 |
