# 問題設定の再設計: LLM活用開発における品質保証の可視化

---

## 1. 再設計した問題設定 -- 「ねらい」セクションの骨子

### 1.1 背景: LLM活用開発の実態と品質保証のギャップ

LLMの急速な発展により、ソフトウェア開発のワークフローは大きく変化しつつある。現在のLLM活用開発の主流は、**荒い要求仕様からLLMに実装コードを直接生成させる**パターンである。この開発スタイルでは、評価が「要求から現れる基本機能の動作確認」に偏り、単体テスト・コンポーネントテストのレイヤーが構造的に軽視される傾向がある。

しかし、LLMが生成したコードを実リリースに載せるためには、従来の開発と同等以上の品質保証が求められる。具体的には以下の品質情報が必要である:

- **単体テスト・コンポーネントテストによる品質の担保**
- **カバレッジの把握**（どの範囲がテストされているか）
- **処理構造の理解**（クラス図、処理フロー、コンポーネント構成）
- **各関数の静的テスト結果の確認**

従来のコードベース開発では、これらの情報は開発者がコードを読むことで暗黙的に把握していた。しかしLLM活用開発では、コードの作成者がLLMであるため、**コードを直接読むことによる品質把握が困難になる**という根本的な問題が生じる。

### 1.2 本質的課題: 品質情報の可読性ギャップ

ここで浮上する本質的課題は、**品質保証に必要な情報を、要求作成ロール（非開発者を含む）が可読可能な状態で提示する必要がある**という点である。

LLM活用開発では、要求を書く人がそのまま開発を主導するケースが増えている。この要求作成ロールは必ずしもコードの専門家ではない。にもかかわらず、リリース判断に必要な品質情報（テスト網羅性、処理構造、静的解析結果）は従来コードレベルでしか表現されてこなかった。

すなわち、**LLM活用開発には「品質情報のコード非依存な可視化」という未解決の課題が存在する**。

### 1.3 目的: テスト要求モデルによる品質情報の構造化と可視化

本研究は、**テスト要求モデル（YAML形式の構造化中間成果物）**を導入することで、この品質情報の可読性ギャップを埋める手法を提案する。テスト要求モデルは以下の4つの機能を果たす:

1. **レビュー可能性**: コードを読めない人でも「何がテストされているか」を確認できる
2. **トレーサビリティの可視化**: 要求 → テスト要求 → テストコードの追跡関係を明示する
3. **要求レベルの網羅性説明**: 「分岐の80%がカバー」ではなく「99件のテスト要求のうちXX件がカバー」という、要求の言葉で網羅性を説明する
4. **静的解析結果の構造化**: コード非専門者にも理解可能な形で品質情報を構造化する

### 1.4 新規性の再定位

先行研究（CodaMosa, ChatUniTest, TestPilot等）は「LLMによるテスト生成の精度向上」を主眼としている。本研究の焦点はそこではなく、**LLM活用開発における品質保証情報を、コード非専門者を含むステークホルダーに対して可視化・構造化する手法**の提案にある。テスト要求モデルは単なるテスト生成の中間成果物ではなく、**品質保証のコミュニケーション基盤**として位置づけられる。

## 2. 論文全体の再構成案

### セクション1: ねらい

**核心メッセージ**: LLM活用開発では品質保証に必要な情報がコードの中に閉じ込められており、要求作成ロール（非開発者含む）がリリース判断に必要な品質情報にアクセスできない。テスト要求モデルを導入することで、品質情報をコード非依存に構造化・可視化する。

構成:
- 1.1 LLM活用開発の現状と品質保証の構造的課題
  - 荒い要求 → コード生成 → 基本機能動作確認、という評価の偏り
  - 単体テスト・コンポーネントテスト層の軽視
- 1.2 実リリースに必要な品質情報と可読性ギャップ
  - 従来は開発者がコードを読んで把握していた品質情報
  - LLM活用開発では要求作成ロールがこの情報にアクセスする必要
  - コード非専門者向けの品質情報表現が未確立
- 1.3 提案: テスト要求モデルによる品質情報の可視化
  - 4つの機能（レビュー可能性、トレーサビリティ、要求レベル網羅性、静的解析構造化）
- 1.4 新規性と先行研究との差異
  - テスト生成精度の向上ではなく、品質保証の可視化・コミュニケーション基盤の提供

### セクション2: 実施概要

**核心メッセージ**: テスト要求モデルを中心に据えた3フェーズ手法により、コード構造の解析結果を要求レベルの品質情報に変換する。

構成:
- 2.1 手法の全体構成（3フェーズ）
  - Phase 1: リポジトリ解析 -- コード構造の把握と対象選定
  - Phase 2: テスト要求モデル生成 -- コードの品質情報を構造化YAMLに変換
  - Phase 3: テストコード生成 -- テスト要求に基づく検証コードの自動生成
- 2.2 テスト要求モデルのスキーマ設計
  - 5種別（BR/EC/BV/ER/DP）による体系的な品質要求の分類
  - ID体系、優先度、ソースコード参照の構造
  - **強調点**: YAMLの各フィールドが「コード非専門者にとって何を意味するか」の説明
- 2.3 実験設計
  - OSSプロジェクト（sakura-editor）、8関数、約385行
  - 3方式比較の枠組み

### セクション3: 実施結果

**核心メッセージ**: テスト要求モデルは品質情報の可視化基盤として有効に機能し、99件のテスト要求による要求レベルの網羅性管理、トレーサビリティの完全確保、テスト失敗時の迅速な原因特定を実現した。

構成:
- 3.1 品質情報の構造化結果
  - 99件のテスト要求を5種別で体系的に定義
  - **新しい観点**: 「8関数の品質保証に必要な情報が99件の可読な要求に集約された」
- 3.2 要求レベルの網羅性
  - テスト要求カバー率100%の意味: 設計した品質基準の全てに対応するテストが存在する
  - **新しい観点**: 「カバレッジ80%」ではなく「99件の品質要求が全て検証済み」という表現
- 3.3 テスト生成とトレーサビリティ
  - 145件のテスト生成、91.7%初回PASS
  - 要求ID ↔ テストケースの双方向追跡
- 3.4 テスト要求モデルによる障害分析の加速
  - 12件のFAILをテスト要求モデルに照らして迅速に分類・修正
  - **新しい観点**: テスト要求モデルが品質問題の診断ツールとして機能
- 3.5 既存テストとの比較（可視性の観点から）
  - 既存テスト45件: テスト意図が暗黙的、コード読解が必要
  - 提案方式145件: テスト意図がYAMLで明示、コード非専門者でもレビュー可能
- 3.6 三方式の品質情報可視性比較（表の再構成）

### セクション4: 結論

**核心メッセージ**: テスト要求モデルは、LLM活用開発において品質保証情報をコード非専門者にも可読な形で構造化する有効な手段である。これはLLM活用開発の実用化に向けた品質保証プラクティスの基盤となる。

構成:
- 4.1 まとめ: 品質情報の可視化という観点からの成果整理
- 4.2 実務への示唆: LLM活用開発チームにおけるテスト要求モデルの活用場面
- 4.3 一般化可能性: 5種別の言語非依存性、他プロジェクトへの展開可能性
- 4.4 今後の課題: コードカバレッジとの相関、副作用関数への拡張、要求作成ロールによるレビュー実験

## 3. 既存実験データの再解釈

### 3.1 「99件のテスト要求」の再解釈

**旧解釈**: 8関数のホワイトボックステスト要求を5種別で体系的に定義した。

**新解釈**: 8関数（約385行）の品質保証に必要な情報が、**コード非専門者でも理解可能な99件の構造化された要求**に集約された。これは「約385行のC++コードを読む」という行為を、「99件のYAMLエントリをレビューする」という行為に変換したことを意味する。

**具体的なインパクト**: 各テスト要求には日本語の説明、優先度、根拠が付与されており、例えば「BR-20: 修飾子alphaの完全一致分岐」という記述は、C++のif文を読まなくても「バージョン文字列中のalphaという修飾子を正しく認識するテスト」として理解可能である。

### 3.2 「145件のテスト、91.7%初回PASS」の再解釈

**旧解釈**: LLMのテスト生成精度が高い（91.7%が正しい期待値を推論）。

**新解釈**: テスト要求モデルが存在することで、12件の失敗が**要求レベルで即座に分類・診断可能**であった。テスト要求モデルがなければ、145件のテストコードとC++ソースコードを突き合わせて原因を特定する必要があった。テスト要求モデルは「何が間違っているか」の診断を、コードレベルから要求レベルに引き上げた。

**具体的なインパクト**: 12件のFAILは5つの原因分類に整理されたが、この分類自体がテスト要求モデルの種別（BR/EC/BV等）に沿って構造化されている。「ParseVersionのコンポーネント分割ロジックの誤解」はBR-20〜23に対応し、コード非専門者でも「バージョン解析の修飾子処理部分に問題がある」と理解可能である。

### 3.3 「テスト要求カバー率100%」の再解釈

**旧解釈**: テスト生成の網羅性が高い。

**新解釈**: 品質保証の説明責任を果たせる状態が達成された。「このコードは十分にテストされているか?」という問いに対し、「99件の品質要求（分岐55件、同値クラス27件、境界値11件、エラーパス3件、依存切替3件）の全てに対応するテストが存在し、全てPASSしている」と**コードを見せることなく説明可能**である。

### 3.4 「既存テスト45件 vs 生成テスト145件（3.2倍）」の再解釈

**旧解釈**: テスト数が3.2倍に増加した。

**新解釈**: 既存テスト45件は「何をテストしているか」がテストコードの中にしか存在しない。生成テスト145件は全てテスト要求IDを持ち、「何をテストしているか」がYAMLの中に明示されている。数の増加以上に重要なのは、**テストの意図が外部化・構造化された**ことである。

さらに、既存テストでカバーされていなかった要求（BR-20〜23, BV-07, EC-17, DP-02, DP-03等）が発見されたが、これは「テスト要求モデルが品質のギャップを可視化した」結果である。テスト要求モデルがなければ、これらのギャップは認識すらされなかった。

### 3.5 「5種別（BR/EC/BV/ER/DP）の分布」の再解釈

**旧解釈**: 分岐網羅55件(55.6%)が最多。ホワイトボックステストとして妥当な分布。

**新解釈**: 5種別は品質情報の「観点」を構造化したものであり、コード非専門者に対して「このコードの品質はどのような観点から検証されているか」を説明する枠組みとなる。

- **BR（分岐網羅）55件**: 「コード内の全ての分岐判断が正しく動作することを確認」
- **EC（同値クラス）27件**: 「入力データの各カテゴリに対して正しく処理されることを確認」
- **BV（境界値）11件**: 「入力の境界（上限・下限・切り替わり点）で正しく動作することを確認」
- **ER（エラーパス）3件**: 「異常な入力に対して安全に処理されることを確認」
- **DP（依存切替）3件**: 「関連する機能間で整合性があることを確認」

この5観点の説明は、C++の知識がなくても品質保証の構造を理解可能にする。

### 3.6 「トレーサビリティマトリクス」の再解釈

**旧解釈**: テスト要求→テストケースの正方向・逆方向追跡が可能。

**新解釈**: トレーサビリティは品質保証の**説明責任の基盤**である。「要求Aは本当にテストされているか?」「このテストは何のために存在するか?」という問いに、コードを読むことなく即座に回答可能である。これはLLM活用開発において、要求作成ロールがテスト成果物をレビューする際の必須基盤となる。

## 4. 追加すべき分析

### 4.1 テスト要求モデルの「可読性」の定量評価

**分析内容**: テスト要求モデルの各エントリについて、理解に必要な前提知識のレベルを分類する。

- **Level 1（コード知識不要）**: 「空文字列の書式処理」「標準的なメールアドレスの判定」等
- **Level 2（ドメイン知識のみ必要）**: 「バージョン文字列中のalpha修飾子の認識」等
- **Level 3（コード知識が必要）**: 「null文字によるループ脱出」「3桁目での数値切り捨て」等

**既存データからの導出**: 99件のテスト要求のdescriptionフィールドを上記3レベルで分類し、Level 1+2の割合を「コード非専門者可読率」として報告する。

**期待される知見**: テスト要求の多くがLevel 1-2に分類され、テスト要求モデルがコード非専門者にとっても実質的に可読であることを示す。

### 4.2 テスト失敗診断における要求モデルの効果

**分析内容**: 12件のテスト失敗について、テスト要求モデルの有無による診断プロセスの差異を比較する。

- **テスト要求モデルあり**: テスト要求ID → 該当する要求の説明 → 期待値の根拠確認 → 修正
- **テスト要求モデルなし**: テストコード → 対象ソースコード → 分岐条件の追跡 → 期待値の再計算 → 修正

**既存データからの導出**: 12件のFAILの原因分類（5分類）がテスト要求モデルの種別と対応付けられることを示し、診断の構造化効果を論証する。

**期待される知見**: テスト要求モデルが障害診断を「コード読解」から「要求照合」に変換し、コード非専門者でも障害の性質を把握可能にしたことを示す。

### 4.3 品質情報の圧縮率

**分析内容**: ソースコード行数に対するテスト要求数の比率を算出し、「品質情報の圧縮率」として提示する。

**既存データからの導出**:
- 約385行のC++コード → 99件のテスト要求
- 圧縮率: 約3.9行/要求
- 領域別: format.cpp（約115行/46要求 = 2.5行/要求）、CWordParse.cpp（約180行/34要求 = 5.3行/要求）、convert_util.cpp（約90行/19要求 = 4.7行/要求）

**期待される知見**: コードの複雑度（分岐密度）と品質情報の圧縮率の関係を考察し、テスト要求モデルがコードの複雑度に比例した適切な粒度で品質情報を構造化していることを示す。

### 4.4 既存テストの品質ギャップの可視化効果

**分析内容**: テスト要求モデルによって発見された「既存テストの未カバー要求」を体系的に整理し、テスト要求モデルの品質ギャップ発見能力を論じる。

**既存データからの導出**: 既にevaluation-reportで特定されている未カバー要求（BR-20〜23, BV-07, EC-17, DP-02, DP-03等）を、5種別ごとに集計する。

**期待される知見**:
- 既存テストは主にBR（分岐網羅）の主要パスをカバー
- EC（同値クラス）の網羅的検証、BV（境界値）の体系的検証、DP（依存切替）のテストは既存テストに欠落
- テスト要求モデルは品質ギャップを**種別ごとに構造化して可視化**する能力を持つ

### 4.5 テスト要求の種別バランスと品質特性の関係

**分析内容**: BR:EC:BV:ER:DPの比率（55:27:11:3:3）が、対象コードの品質特性（分岐密度、入力多様性、エラー処理の充実度）とどう関係するかを分析する。

**既存データからの導出**: 関数ごとの種別分布を比較し、関数の性質との対応を論じる。

**期待される知見**:
- 分岐密度の高い関数（ParseVersion等）はBR比率が高い
- 入力パターンが多様な関数（IsMailAddress等）はEC比率が高い
- ER/DPが少ないのは対象関数が純粋関数であることに起因し、副作用を持つ関数ではER/DP比率が増加すると予測される
