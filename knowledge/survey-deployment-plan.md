4. 共有URLから ID を抽出（`https://drive.google.com/file/d/XXXXXX/view` の `XXXXXX`）
5. **以下2つの方法から選択**:

   **A. 既存 Form に画像を手動追加**（既存URLを維持）
   - 上記編集URLで Form を開く
   - 各セクションに「+」→「画像」→「ドライブから選択」で5枚挿入
   - Q26-Q29 がまだ無ければ手動で追加（survey-design.md §3 [5] 参照）

   **B. GAS で Form を再生成**（推奨・再現性高い）
   - `scripts/create_survey_form.gs` の `SURVEY_IMAGES` に ID を設定
   - `createSurveyForm()` を実行 → 画像込み33問の Form を新規生成
   - 既存 Form は保管用として残し、新 Form を本番運用

## 2. パイロットテスト（本展開前の検証）

### 2.1 パイロット参加者

- **対象**: チーム内3名（エンジニア2 + 非エンジニア1が理想）
- **目的**: 質問の明瞭性 / 所要時間 / 技術的不具合を発見

### 2.2 パイロット前の合意事項

- 研究メンバー同士の実施であることを明示
- 統計データには本パイロット結果は含めない（本展開でクリーン回答を収集）
- フィードバックは「質問文が曖昧か」「時間超過か」「画像が表示されるか」に集中

### 2.3 パイロット評価項目

| 観点 | チェック内容 | 許容値 |
|---|---|---|
| 所要時間 | 開始〜送信までの実測 | 12〜15分以内 |
| 質問の明瞭性 | 「意味がわからない」とマークされた問 | 0件（あれば文言修正） |
| 表示の整合性 | 画像・コード・絵文字が正しく描画されるか | 全件正常 |
| ブラウザ差異 | Chrome / Safari / mobile での動作 | 全OK |

### 2.4 パイロット後のアクション

- Q18 / Q24 の自由記述コメントを精読し、質問文の改善点を抽出
- 所要時間が15分を超える場合 → 設問の削減検討
- パイロット3名ぶんのデータは削除または「テスト」ラベルで隔離

## 3. 本展開

### 3.1 展開チャネルと目標回答数

| チャネル | 目標回答数 | 属性の偏り | 特徴 |
|---|---|---|---|
| ヒトハブテックフェス等イベント | 15〜30 | エンジニア寄り | 対面で補足説明可能、当日回収 |
| 社内・関係者DM | 10〜15 | 意図的に非エンジニア多め | 層別調整の主戦力 |
| SNS（X, 社内Slack, LinkedIn） | 10〜50 | 属性変動大 | 量の確保・外部妥当性 |
| **合計目標** | **35〜95** | 非エンジニア比率 ≥ 40% | |

最低ライン: **合計30回答、非エンジニア12名以上** をクリアすれば分析可能。

### 3.2 リクルート用メッセージ（テンプレ）

#### 3.2.1 汎用（社外向け）

> 【5〜10分アンケートにご協力ください】
>
> ソフトウェア開発の「品質情報を非エンジニアにも伝わる形式で共有できるか」という研究を進めています。
>
> コードを読んだ経験がなくても問題ありません。むしろ **普段コードを書かない方の回答が貴重** です。
>
> 内容: リバーシの「石を置けるか判定」するコードを題材に、複数の説明形式のわかりやすさを評価いただく
>
> 所要時間: 約12〜15分
>
> 回答URL: {Published URL}
>
> 結果は学術発表（SQiP 2026等）で匿名集計のみ使用します。

#### 3.2.2 非エンジニア向け特別版

> 【エンジニア以外の方へ、5〜10分のアンケートのお願い】
>
> ソフトウェアの「テスト要求」という設計情報を、**企画・デザイン・営業などコードを書かない方でも読めるか** を検証しています。
>
> 「わからなかった」という回答も非常に大切なデータです。
>
> 内容: リバーシというシンプルなゲームのプログラムをテーマに、3種類の説明図のどれが最もわかりやすいかをお尋ねします。
>
> 回答URL: {Published URL}

#### 3.2.3 エンジニア向け対照群

> 【エンジニアの皆様へ、対照群データご協力のお願い】
>
> 非エンジニアへの伝達妥当性を検証する研究で、エンジニア回答との比較データを必要としています。
>
> 普段読まれているテスト設計との比較観点で、ご回答いただけると嬉しいです。
>
> 回答URL: {Published URL}

## 4. 回答データ分析プロトコル

### 4.1 基本集計（定量）

Google Sheets で以下の pivot / 関数を用意:

| 指標 | 計算 | 対応Q |
|---|---|---|
| 回答者属性分布 | COUNTIF 各属性 | Q1/Q2/Q5 |
| パターン別平均理解度 | AVERAGE of Q9/Q11/Q13 | Q9/Q11/Q13 |
| パターン選好率 | COUNTIF per choice / total | Q15/Q16/Q17 |
| 合成→実コード理解度変化 | AVERAGE of Q20 | Q20 |
| 総合評価 | AVERAGE of Q22/Q23 | Q22/Q23 |

### 4.2 層別分析（重要）

以下のカットで再集計:

1. **Q5 立場別**: 非エンジニア / 技術系PM / エンジニア
2. **Q2 経験年数別**: 3年未満 / 3-5年 / 5年以上
3. **Q3 頻度別**: 読まない / 月数回 / 週数回以上

各カットで Q15 (最も理解しやすい) の分布を比較。仮説 H2 の検証に使用。

### 4.3 仮説検定（R または Python）

| 仮説 | 検定手法 | ツール |
|---|---|---|
| H1: 視覚化が最多選好 | カイ二乗検定 | scipy.stats.chisquare |
| H2: 経験年数×選好の相関 | クロス表独立性検定 | scipy.stats.chi2_contingency |
| H3: 合成→実コード理解度維持 | 対応ありt検定 or Wilcoxon | scipy.stats.wilcoxon |
| H4: 階層化で十分性体感向上 | ANOVA | scipy.stats.f_oneway |

### 4.4 定性分析

Q18 / Q24 の自由記述を以下の手順で整理:

1. 全コメントをスプレッドシートに転記
2. KJ法的にカテゴリ化（以下の候補から開始）
   - 「図がわかりやすかった」系
   - 「技術用語が難しかった」系
   - 「もっと情報が欲しい」系
   - 「改善案」系
3. カテゴリ別に頻度集計
4. 代表コメント3-5件を引用候補として選定

## 6. リスクと対策

| リスク | 可能性 | 対策 |
|---|---|---|
| 目標回答数に達しない | 中 | SNS展開、追加期間、属性別リクルート強化 |
| 非エンジニア回答が少ない | 高 | 社外コミュニティへの依頼、展開期間延長 |
| Forms が Google 認証エラー | 低 | 代替として Typeform を用意 |
| 題材が難しすぎて大量離脱 | 中 | パイロットで所要時間・離脱率を確認、簡略化 |
| Q8 検証問題で「コードを読まず回答」が検出 | 中 | 該当回答を統計から除外 |
| SNS で炎上（設問が差別的など） | 低 | パイロット時に表現を慎重にレビュー |
