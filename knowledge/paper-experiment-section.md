## 4.1 実験対象の選定方針

本研究の目的は、コードのみを入力として (1) ホワイトボックス評価における確認対象領域の十分性を高めること、(2) 非エンジニアにも伝わる形で品質情報を共有できること、の2点の妥当性を問うことにある。この問いに適う実験対象の選定には、次の要件を課した。

1. **実在性**: 研究用に作成した合成コードではなく、実際に保守されているOSSを対象とする。実務適用可能性を示す外的妥当性を確保するためである。
2. **多様性**: 単一言語・単一ドメインでの成立では、本手法がAI駆動開発一般に適用可能であることの示唆としては弱い。したがって、少なくとも2つの異なる言語・ドメイン・設計スタイルを選定する。
3. **可観測性**: 対象リポジトリに既存単体テストが存在すること。これにより、提案手法で導出した要求が既存テストにどの程度反映されているか、また逆に既存テストで見落とされていた観点を新たに検出できたかを定量的に比較可能となる。
4. **適切な規模**: 総行数が数万行規模かつ対象モジュールが1万行以下であること。これは論文期間内に評価軸（`本稿 §5`）のすべてを算出可能とするためである。
5. **非属人性**: 実証対象が特定の研究者や組織のコードでないこと。再現可能性と査読時の中立性を確保する。

## 4.2 選定結果

上記要件を踏まえ、以下の3対象を選定した。

| 対象 | 言語 | 規模（対象モジュール） | 役割 |
|---|---|---|---|
| sakura-editor/sakura | C++ | 約385行 / 8関数 | 主実証。手法の提案・実証のベースライン |
| pallets/click | Python | 約2000行 / 対象クラス群 | 汎用性検証。言語・ドメイン・OOP設計スタイルの差異下でも手法が成立することを確認 |
| colinhacks/zod（任意） | TypeScript | 約3000行 / 対象クラス群 | 拡張性検証。型ガード・判別共用体等の言語固有パターンに対する本手法のv3.0拡張の有効性を確認 |

sakura-editorはC++のテキストエディタであり、書式処理・文字種判定・全角半角変換という複数ドメインの純粋関数群を持つ。pallets/clickはPythonのCLIフレームワークであり、抽象基底クラス・デコレータ・コンテキストマネージャという動的型言語特有のOOPパターンを含む。colinhacks/zodはTypeScriptのスキーマ検証ライブラリであり、判別共用体・型ガードといった型システム駆動の設計を特徴とする。これら3対象は、言語パラダイム（静的型/動的型/段階的型）・設計スタイル（関数中心/OOP中心/型中心）・ドメイン（テキスト処理/CLI構築/検証）の三軸で互いに異なる。

## 4.3 非エンジニアへの情報提供妥当性の検証方針

研究の問い (2) に対しては、上記3対象とは別に、非エンジニアを含む回答者への **アンケート調査** を実施する。アンケートでは、実在OSSそのものを素材として扱うと回答者の前提知識不足で無効回答が増えるため、**題材を最小化した合成サンプル**（リバーシの合法手判定処理、約80行）を用意し、これを複数のテスト要求モデル提示形式（自然文・階層化・視覚化併用）でレンダリングし、回答者に可読性・理解度・好感度を評価してもらう。その上で、sakura-editorから抜粋した実サンプルを追加提示し、合成サンプルでの可読性が実OSSに対してどの程度維持されるかを観察する。これにより、研究の問い (2) を**合成と実在の両条件下で** 検証する。

## 4.4 click 実証結果（追実験・2026-04-21 実施）

第2実証対象として `pallets/click` の `src/click/types.py`（1,209行、19クラス）から ParamType 継承階層の代表5クラス（`ParamType`, `StringParamType`, `Choice`, `_NumberParamTypeBase`, `IntRange`）を選定し、本手法 v3.1 を適用した。その結果、131件のテスト要求を導出した。内訳は BR=37, EC=18, BV=11, ER=9, DP=5, CI=18, SV=9, CP=9, EN=15 である。sakura-editor実証の99件（関数4つ、約385行）と同規模のコード量で約32%多い要求密度となったが、従来5種別（BR/EC/BV/ER/DP）に限った件数は click が80件と sakura の99件より少ない。差の51件は v3.1 の OOP拡張（CI/SV/CP/EN）が Python OOP 特有の設計観点を捕捉した分に該当し、手法の汎用性を示す。

特に本研究で新設した EN（カプセル化）種別 15件は、`Choice.choices` の public mutable state、`_NumberParamTypeBase._number_class` の派生設定契約、`IntRange` のダイヤモンド継承下での構築契約、`FloatRange` との clamp+open_bound 許容差異など、**CI/SV では明示化されにくい構造設計観点** を抽出した。カプセル化リスクとして9件（重大度 high=3, medium=4, low=2）が検出され、これらは既存テスト（`tests/test_types.py` の11関数）では検証されていない観点であった。

`tests/test_types.py` との網羅比較では、本 TRM 173件（TGT-06/07/08 を含めた拡張版、ClickException階層とmake_pass_decorator含む）のうち既存テストでカバーされている観点は **約 5.8%（10件）** であり、残る **約 86.7%（150件）が既存テストに存在しない追加観点** であった。種別別の GAP 率は EN 95%, SV 100%, DP 100%, CI 92%, CP 93% で、構造設計観点の大半が既存テストで未検証であることが定量的に確認された。本手法は既存テストを補完する形で品質保証の観点を拡張する効果があり、これは研究の問い (1)「ホワイトボックス評価における確認対象領域の十分性の向上」への具体的裏付けとなる。

## 4.6 可視化レイヤ（非エンジニア向け提示形式）

研究の問い (2)「非エンジニアにも伝わる形で品質情報を共有できるか」に対する手法としては、TRM YAML を入力として **Sunburst / Sankey / Chord / Heatmap の4種の図式** を自動生成する可視化レイヤを追加で提案する。実装は `scripts/generate_visualizations.py` として提供し、各 TGT について以下を出力する:

- **Sunburst (Mermaid mindmap)**: テスト要求を種別別に階層化した全体俯瞰図
- **Sankey (Mermaid sankey-beta)**: 種別分布と優先度の流量を示すフロー図
- **Heatmap Matrix (Markdown table)**: 状態変数 × カプセル化リスクの交差表。設計上の懸念が定量化されて可視化される
- **Chord 風 Flowchart (Mermaid flowchart)**: 状態変数と検出リスクの相互関係を線種（実線/破線/点線）で重大度別に表現

この可視化レイヤの役割は、TRM の詳細な要求リスト（click の場合 131〜173件）をそのまま非エンジニアに提示せず、**情報の抽象度を段階的に下げた 4種類の視覚表現** を経由して共有することにある。`complex-scope-approaches.md` で整理した階層化パターン（A）と視覚化パターン（D）を統合した実装形態であり、手続き的に自動生成されるため、パイプライン上の追加コストは軽微である。

3Dグラフは検討したが、離散的なトリガー組合せの可視化においては遮蔽・遠近歪み・静的媒体不適合の観点から 2D 手法に劣ると判断し、本研究では採用していない。
