# 複雑化範囲へのTRMアプローチ（複数パターン案）

> **目的**: コード上の"複雑な範囲"（長大関数・深い継承・状態マシン・非同期/並行処理等）に対して、TRMがどのようにアプローチすべきかを複数パターンで示す
> **起点**: input.md ②「複雑化した範囲についてどのようにアプローチをするのかについてTRMのアプローチを考える → ユーザにわかるようなアウトプットを作成する → 目的はより堅牢なソフトウェアを開発する」
> **担当案**: 八木さん主導。本ドキュメントは末竹側からの選択肢提示として機能
> **関連**: `./trm-evaluation-criteria.md`（評価軸）、`./architecture-diagram.md`（現在のTRM構造）

---

## 1. 「複雑化範囲」の定義

TRMが対応に苦慮する典型的なコード特性を以下に整理する:

| 複雑性の型 | 具体例 | 現TRMでの扱いの弱点 |
|---|---|---|
| C1. 長大関数 | 200行以上の単一関数 | BRが過多になりユーザーが俯瞰できない |
| C2. 深い分岐ネスト | if-elseの6段以上のネスト | BRが組合せ爆発、可読性Level 3化 |
| C3. 状態マシン | ステートパターン・複数遷移ルール | SVだけでは遷移網羅が伝わりにくい |
| C4. 深い継承階層 | 5段以上の継承・多重継承 | CIだけでは階層全体像が見えない |
| C5. 非同期/並行 | コールバック・Promise・スレッド | 現スキーマは明示的観点が少ない |
| C6. 外部依存密 | DB・API・ファイルシステム | DPだけでは振る舞い組合せが膨大 |
| C7. ドメインロジック密 | 帳票計算・税計算・ゲームルール | 仕様書⇔コードのギャップが大きい |
| C8. メタプログラミング | マクロ・デコレータ・テンプレート展開 | CPは存在するが多段展開の表現が弱い |

## 2. アプローチ候補（5パターン）

### パターンA: 階層化TRM（High-level → Detail）

**考え方**: 複雑範囲を要約レイヤ（Summary Requirement）と詳細レイヤ（Detail Requirement）の2階層で構造化。ユーザーは要約だけを見て目的を把握できる。

```
Summary: 「税計算が法制度に従って正しく動作すること」
  ├── Detail: BR-01 軽減税率適用分岐
  ├── Detail: BR-02 非課税判定分岐
  └── Detail: BV-03 区分境界（8% / 10%）の閾値検証
```

**スキーマ拡張案**:
```yaml
test_requirements:
  - id: "SUM-01"
    type: "summary"
    description: "税計算が法制度に従って正しく動作する"
    children: [BR-01, BR-02, BV-03]
    readable_for: ["non-developer", "domain-expert"]
```

**長所**
- 非開発者は Summary だけで理解可能（Level 1 の比率を人為的に担保）
- 詳細はトレース可能で監査性維持

**短所**
- Summary の質がLLM推論に強く依存
- 2階層化で保守コストが増える

**適用ドメイン**: C1, C7

---

### パターンB: スライス分割（Semantic Slicing）

**考え方**: 長大関数を意味単位でスライスし、スライス毎にTGTを割り当てる。スライスは"前処理/コア処理/後処理"や"バリデーション/計算/出力"等の意味的ブロック。

```
TGT-01: calculateOrder (280行)
  ↓ スライス分割
TGT-01a: 入力バリデーション（45行, BR x5, EC x3）
TGT-01b: 価格計算コア（120行, BR x12, BV x6, SV x4）
TGT-01c: 税率適用（60行, BR x8, BV x4）
TGT-01d: 出力フォーマット（55行, EC x5）
```

**スキーマ拡張案**: TGTに `slice_of` フィールドを追加し、親TGTとの関係を保持。

**長所**
- 長大関数を変更せずにテスト観点を分割可能
- 要求数の組合せ爆発を緩和

**短所**
- スライス境界の同定がLLM依存（誤ったスライスは誤解を生む）
- スライス横断の不変条件を見落とす可能性

**適用ドメイン**: C1, C2

---

### パターンC: シナリオ駆動型TRM（Scenario-First）

**考え方**: コード構造から要求を導くのではなく、**ユースケース/ユーザーシナリオ** を起点にTRMを組む。既存のBR/EC/BV等はシナリオに紐づける。

```yaml
scenarios:
  - id: "SC-01"
    actor: "注文者"
    precondition: "カート商品あり"
    trigger: "チェックアウトボタン押下"
    expected_outcome: "税込合計が表示される"
    involves_requirements: [BR-01, BR-02, BV-03, SV-04]
```

**長所**
- 非開発者（プロダクトオーナー等）が最も理解しやすい形式
- 要求変更時の影響範囲が「シナリオ単位」で見える

**短所**
- シナリオ網羅性の保証が別途必要（シナリオ自体が漏れる）
- コードレベルの細かい分岐との対応が疎

**適用ドメイン**: C7, C6

---

### パターンD: 視覚化レイヤ追加（Visual Layer）

**考え方**: TRMテキストと並置する形で、**状態遷移図・シーケンス図・継承階層図・依存グラフ** を自動生成しMarkdown埋め込み（Mermaid等）。理解負荷を視覚で引き受ける。

```yaml
visual_aids:
  - target: "TGT-03: OrderStateMachine"
    type: "state_diagram"
    notation: "mermaid"
    content: |
      stateDiagram-v2
        [*] --> Pending
        Pending --> Paid : pay()
        Paid --> Shipped : ship()
        Shipped --> Delivered : deliver()
        Pending --> Cancelled : cancel()
```

**長所**
- 状態マシン・継承階層・依存が一瞬で伝わる
- 可読性Level 1相当の理解を視覚で実現
- 既存TRM構造を崩さず付加

**短所**
- 生成品質がLLM依存（構造の誤描画は誤解を生む）
- Mermaid等の環境前提がつく

**適用ドメイン**: C3, C4, C5

---

### パターンE: 段階的監査（Progressive Audit）

**考え方**: 複雑範囲に対しては一度に全要求を生成せず、**「まず骨格 → 監査 → 洗練」** の反復で品質担保する。現 `audit-trm` の強化版。

```
Phase 1: 骨格生成（BR・ECのみ、high priority のみ）
  → 初期 TRM (20要求)
Phase 2: 監査フェーズ1（抽象的な観点の漏れ確認）
  → 追加 30要求
Phase 3: 詳細化（BV・ER・DP、OOP追加）
  → 追加 40要求
Phase 4: 監査フェーズ2（実装と期待値の突き合わせ）
  → 精度確認・修正
```

**長所**
- 各フェーズで人間の介入機会があり、LLM暴走を防げる
- 複雑範囲で特にコスト効率が良い

**短所**
- ワークフローが長くなる
- 各フェーズの終了判定基準が必要

**適用ドメイン**: C1, C2, C6, C7

## 3. 各パターンの適用指針（どれをいつ使うか）

| 複雑性の型 | 推奨パターン | 次点 |
|---|---|---|
| C1. 長大関数 | B: スライス分割 | E: 段階的監査 |
| C2. 深い分岐ネスト | A: 階層化TRM | B: スライス分割 |
| C3. 状態マシン | D: 視覚化レイヤ | A: 階層化 |
| C4. 深い継承階層 | D: 視覚化レイヤ | A: 階層化 |
| C5. 非同期/並行 | D: 視覚化レイヤ | E: 段階的監査 |
| C6. 外部依存密 | C: シナリオ駆動 | E: 段階的監査 |
| C7. ドメインロジック密 | C: シナリオ駆動 | A: 階層化 |
| C8. メタプログラミング | E: 段階的監査 | D: 視覚化 |

**複合適用例**: 長大関数で状態マシンを含むコード（C1+C3）には **B+D**（スライス分割 + 各スライスに状態遷移図を付与）を組み合わせる。

## 4. ユーザー可読アウトプットの形式案

複雑範囲のTRMを最終的にユーザー（非開発者含む）に見せる時の形式案:

### 案1: TRMダッシュボード（Markdown単一ファイル）

```markdown
# TGT-03: 注文状態機械

## 概要（Level 1）
注文の状態が「仮予約→確定→発送→配達完了」と正しく遷移し、
キャンセル操作では適切な状態に戻ること。

## 状態遷移図
[Mermaid図]

## 要求一覧
| ID | 要約 | 詳細 | 可読性 |
|---|---|---|---|
| SUM-01 | 正常遷移の網羅 | [詳細へ](#sum-01) | L1 |
| SUM-02 | キャンセル処理の整合性 | [詳細へ](#sum-02) | L1 |

## 詳細（Level 2-3）
...
```

### 案2: インタラクティブHTML

- 折り畳み可能な階層ツリー
- Mermaid図のリアルタイム描画
- Level 1/2/3 のフィルタ
- 検索（要求ID、キーワード）

実装: Jekyll + カスタムレイアウト / Docusaurus / MkDocs

### 案3: 仕様書形式PDF

- 表紙: シナリオ一覧
- 章: 機能単位（スライス単位）
- 節: 要求一覧（表形式）
- 付録: 低レベル要求・コード参照

**推奨**: まずは **案1（Markdown）** から着手。既存の `test-requirement-model.md` 生成に視覚レイヤ・階層化を追加する形で拡張。

## 5. より堅牢なソフトウェアを開発する、への寄与

「TRMで複雑範囲を扱う」ことがどのように堅牢性に寄与するかを整理:

| アプローチ | 堅牢性への寄与 |
|---|---|
| A: 階層化 | 仕様の意図が失われず伝わる → 実装変更時の退行検出性向上 |
| B: スライス分割 | 部分単位で責務が明瞭 → 部分修正の影響範囲が限定 |
| C: シナリオ駆動 | ユーザー価値に直結した観点の取りこぼし防止 |
| D: 視覚化 | 状態・依存の不整合を図で発見可能 → 仕様バグ検出 |
| E: 段階的監査 | 大規模コードでも漏れなく網羅 → 長期的品質劣化防止 |

## 6. 次の検討ステップ

1. **優先パターンの決定**: 八木さん・末竹で5パターンから1-2個を選定（まず試す対象）
2. **プロトタイプ**: 選定したパターンを既存TRMスキーマ拡張 or 別ファイル生成で試作
3. **検証リポジトリでの適用**: `./candidate-repositories.md` の候補リポジトリで試行
4. **評価**: `./trm-evaluation-criteria.md` のカテゴリBで可読性改善効果を測定
5. **論文/発表への反映**: 有効なパターンを次版論文の提案手法として組み込み

## 7. リスク・未解決項目

| 項目 | リスク | 対応案 |
|---|---|---|
| LLM推論依存の増大 | 階層化・シナリオ化で抽象表現が増え、誤推論混入の余地が広がる | 段階的監査（E）との組み合わせ必須 |
| スキーマ拡張の破壊的変更 | v3.0→v4.0 への移行コスト | 追加フィールドはoptionalで入れ、既存TRMと共存 |
| 視覚レイヤの保守負担 | Mermaid図の手動修正が発生する可能性 | 生成フェーズで再生成可能な構造にする |
| 選定パターンの偏り | 担当者の好みが強く出る | 最低2パターンを複数リポジトリで比較評価 |
