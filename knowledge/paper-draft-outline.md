## Abstract（300字）

LLM を先行させた AI駆動開発が広がる中、実装領域のホワイトボックス品質情報がコードに閉じ込められ、要求作成ロール（非エンジニアを含む）がアクセスできない「可読性ギャップ」が生じている。本研究は、コードのみを入力として (1) ホワイトボックス評価における確認対象領域の十分性を高めること、(2) 非エンジニアにも伝わる形で品質情報を共有できることの2点を妥当性の問いとして立て、テスト要求モデル (TRM) を中間成果物とする手法を提案・実証する。C++手続き型 OSS（sakura-editor）と Python OOP OSS（pallets/click）で計2実証を行い、それぞれ99件・173件の要求を導出、sakura では可読率 65.7% を達成した。v3.1 では新設のカプセル化 (EN) 種別により OOP の構造設計観点を15件抽出し、既存テストのGAP 86.7% を埋める追加観点として機能した。

4. C++手続き型とPython OOPの2実証による汎用性検証

## §3. 提案手法

### 3.1 手法の全体像（`knowledge/architecture-diagram.md` §1 を再利用）

- パイプライン図（analyze → generate-trm → audit → generate-tests → visualize）

### 3.2 TRM v3.1 スキーマ

- 従来5種別 (BR/EC/BV/ER/DP)
- OOP拡張3種別 (CI/SV/CP) — v3.0
- **カプセル化種別 EN** — v3.1 新設（本研究の主要貢献）
- SV の新サブタイプ2種（member_declaration_validity / member_initialization_requirement）

### 3.3 EN (Encapsulation) 種別の設計

#### 3.3.1 5サブタイプ

| サブタイプ | 検証対象 |
|---|---|
| access_control_correctness | メンバ可視性の設計整合性 |
| leaky_accessor | getter/setter の内部状態漏洩 |
| mutability_contract | final/const/readonly の宣言整合性 |
| construction_contract | コンストラクタでの必須field初期化 |
| invariant_surface | public境界でのクラス不変条件維持 |

#### 3.3.2 EN が補完する観点

既存 CI/SV では「動作の正しさ」を検証するが、EN は「構造の設計妥当性」を検証する。具体例:

- `Choice.choices` の public mutable 問題（SVの振る舞いでは検出困難）
- `_NumberParamTypeBase._number_class` の派生設定契約（CIの abstract_coverage より直接的）
- `IntRange` の `min > max` 許容設計の是非

### 3.4 可視化レイヤ

TRM YAML を入力として4種の図式を自動生成:

| 手法 | 目的 | 実装 |
|---|---|---|
| Sunburst | 要求の種別階層俯瞰 | Mermaid mindmap |
| Sankey | 種別×優先度の流量 | Mermaid sankey-beta |
| Heatmap | state × risk の複合影響 | Markdown table |
| Chord | トリガー相互関係・禁忌組合せ | Mermaid flowchart |

3Dグラフは検討したが、離散的なトリガー組合せの可視化では 2D に劣る（遮蔽・歪み・静的媒体不適合）。

## §7. 考察

### 7.1 研究の問い (1) への応答

- 既存テストの GAP 86.7% を本手法が埋める
- v3.1 の EN で構造設計観点が新たに15件検出
- **確認対象領域の十分性は定量的に向上した**

### 7.2 研究の問い (2) への応答

- sakura（手動分類）の可読率 65.7% に対し、click（自動分類）は 45.7%
- OOP固有語彙（MRO, __mro__, ClassVar 等）が L3 を押し上げる
- **可読性は対象ドメインの技術性に強く依存**。純粋関数中心のドメインでは達成しやすいが、内部APIでは低下
- 視覚化レイヤを通せば、非エンジニアが **Summary 層のみを参照する運用** で補完可能

### 7.3 v3.1 の価値と限界

- OOP中心コードで 50%以上の追加観点を捕捉（click 実証）
- 手続き型では効果限定的（sakura: +2.4%）
- → 対象の設計スタイルで v3.0 / v3.1 を使い分ける指針を提示

### 7.4 可視化レイヤの効果

- 4種の図式が Summary / Detail 両方の層を提供
- 3D は採用せず、2D の組合せで十分な情報密度

### 7.5 妥当性への脅威

- 合成題材（リバーシ）と実在OSSの理解度差（アンケートで検証予定）
- ルールベース分類の誤差（LLM分類との比較が必要）
- 単一言語・単一実装者の手書きTRMとの比較（他者による検証が未実施）
