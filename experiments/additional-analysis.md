### 1.1 分類基準

テスト要求99件のdescriptionフィールドを以下の3レベルで分類した。

| レベル | 定義 | 判定基準 |
|--------|------|----------|
| Level 1 | コード知識不要 | ドメイン用語のみで理解可能。「メールアドレス」「バージョン」「全角半角」等の一般的な概念で説明されている |
| Level 2 | ドメイン知識のみ必要 | 特定ドメインの知識（メール形式、バージョン表記規則、日本語文字種等）があれば理解可能 |
| Level 3 | コード知識が必要 | 変数名、コード構造、マッピング規則、ビット演算等のコード固有知識が理解に必要 |

### 1.4 考察

1. **全体の約2/3（65.7%）がコード非専門者にも可読**: テスト要求の過半数はコードを読めない人でも内容を理解できる。これはテスト要求モデルが品質情報の可視化基盤として一定の有効性を持つことを示す。

2. **種別による可読性の差異が顕著**: EC（同値クラス）とER（エラーパス）は可読率が高い（81.5%、100.0%）。これらは入力パターンの分類やエラー処理という、ドメインレベルで自然に表現できる概念であるためである。一方、DP（依存切替）は関数間の関係性を記述するため、コード知識が必要になりやすい（可読率33.3%）。

3. **領域による可読性の差異**: format.cpp（78.3%）とconvert_util.cpp（73.7%）は可読率が高い。これは日時書式変換や全角半角変換という、一般に理解しやすいドメインを扱っているためである。CWordParse.cpp（41.2%）は文字種結合判定というエディタ固有のロジックを扱っており、ECharKind列挙型やマッピング規則などコード固有の概念が多く登場するため可読率が低い。

4. **Level 3要求の改善可能性**: Level 3に分類された34件のうち、description文を改善することでLevel 2に引き上げ可能なものが存在する。例えば「ラテン → CSYM マッピングパスを検証する」を「ラテン系文字がアルファベットと同種として扱われることを検証する」と記述すれば、コード知識なしで理解可能になる。

### 2.5 考察

1. **圧縮率2.5〜5.0行/要求の範囲**: 大半の関数で「コード2.5〜5行ごとに1つのテスト要求」という粒度になっている。これはコード非専門者がYAMLを読む際の認知負荷として適切な粒度と考えられる。

2. **分岐密度とテスト要求密度の相関**: GetDateTimeFormat（2.5行/要求）やParseVersion（3.0行/要求）は分岐密度が高く、テスト要求密度も高い。テスト要求モデルがコードの複雑度に比例した粒度で品質情報を構造化していることを示す。

3. **CompareVersionの特異性**: 5行のコードに対して6件のテスト要求がある（0.8行/要求）。これは薄いラッパー関数でもテスト観点（等価性、大小関係、修飾子順序、区切り文字等価性、依存関数統合）が多様であることを示している。

### 4.3 考察

1. **DP（依存切替）のギャップが最大**: 既存テストでは関数間の関係性を検証するテスト（往復変換、通常版vs4KW版の差異比較、ParseVersionの間接検証）が完全に欠落している。DPテストは「関数の責務を超えた品質観点」であり、手動テストでは設計意図がなければ作成されない種類のテストである。

2. **BV（境界値）のギャップが大きい**: 既存テストでは境界値の体系的な検証が行われていない（未カバー率72.7%）。特にIsValidCharの文字コード境界（0x20/0x21/0x7E/0x7F）や全角文字範囲の境界は、off-by-oneエラーの検出に重要だが、既存テストではカバーされていなかった。

3. **EC（同値クラス）のギャップも大きい**: 未カバー率70.4%。既存テストは主要な正常系をカバーしているが、入力パターンの体系的な分類（空文字列、末尾%、修飾子順序、区切り文字バリエーション等）は行われていなかった。

4. **BR（分岐網羅）は相対的にカバーされている**: 未カバー率52.7%と他種別より低い。既存テストは主要な分岐パス（正常系の分岐）をカバーしているが、コーナーケース（修飾子の部分一致、3桁目数値切り捨て等）の分岐は未カバーである。

5. **テスト要求モデルによるギャップ可視化の効果**: 上記の分析自体が「テスト要求モデルが存在するからこそ可能」である。テスト要求モデルがなければ、既存テストの品質ギャップは「テストが少ない」という漠然とした認識にとどまり、「何の観点が欠けているか」を構造的に把握することはできなかった。

## 総括

### 主要な定量的知見

| 指標 | 値 |
|------|-----|
| コード非専門者可読率 | 65.7%（65件/99件） |
| 品質情報の圧縮率 | 3.9行/要求（平均） |
| 既存テスト未カバー率 | 61.6%（61件/99件） |
| テスト失敗のうち要求レベルで診断可能な割合 | 100%（12件/12件） |
| テスト失敗のうちコード非専門者が理解可能な割合 | 約67%（8件/12件） |

### 論文への示唆

1. **品質情報の可視化効果は関数の性質に依存する**: 日時書式変換や全角半角変換のような「一般に理解しやすいドメイン」の関数ではコード非専門者可読率が73〜78%に達するが、文字種結合判定のような「エディタ固有のロジック」では41%にとどまる。テスト要求モデルの可視化効果はドメインの一般性に相関する。

2. **種別（BR/EC/BV/ER/DP）は品質ギャップの構造化に有効**: 既存テストのギャップはDP（100%未カバー）> BV（72.7%）> EC（70.4%）> ER（66.7%）> BR（52.7%）の順で大きい。この構造化された知見は「何が足りないか」をコード非専門者にも説明可能にする。

3. **テスト失敗の診断抽象度が大幅に向上**: テスト要求モデルにより、診断の起点が「16進数のテスト出力」から「日本語のテスト要求説明」に引き上げられた。12件全てが要求レベルで分類可能であり、うち8件はコード非専門者にも理解可能な粒度で診断できた。

4. **品質情報の圧縮率は実用的な範囲**: 385行のC++コードが99件のテスト要求に圧縮される（3.9行/要求）。これは「コードを読む」行為を「構造化された要求リストを読む」行為に変換したことを意味し、認知負荷の大幅な削減が期待できる。
