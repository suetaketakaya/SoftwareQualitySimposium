# 手書きTRM vs パイプライン生成TRM 比較レポート

> **対象**: リバーシサンプル (`experiments/reversi/reversi_move.py`)
> **比較元**: `../../knowledge/survey-design.md` §5.3 の手書きTRM（9要求）
> **比較先**: `./test-requirements.yaml` パイプライン生成TRM（45要求）
> **実施日**: 2026-04-21

---

## 1. サマリ

| 指標 | 値 |
|---|---|
| 手書き要求数 | 9 |
| パイプライン生成要求数 | **45** |
| 生成倍率 | **5.0×** |
| 手書き要求のカバー率 | **100%** (9/9) |
| パイプライン側の追加要求 | 36 |
| 手書き側にあり生成側に欠落 | **0** |

**結論**: パイプラインは手書きTRMを**完全に包含**（superset）する。欠落はない。ただし同じ対象コードに対して5倍の粒度で要求を生成しており、この粒度差が品質 or ノイズのどちらに寄与するかは後続で評価すべき論点。

## 2. 型別要求数の比較

| 種別 | 手書き | パイプライン | 比率 |
|---|---|---|---|
| BR (分岐網羅) | 4 | 20 | 5.0× |
| EC (同値クラス) | 1 | 8 | 8.0× |
| BV (境界値) | 2 | 9 | 4.5× |
| ER (エラーパス) | 1 | 5 | 5.0× |
| DP (依存切替) | 1 | 4 | 4.0× |
| **合計** | **9** | **45** | **5.0×** |

- 最も差が大きいのは **EC（同値クラス）** の8倍。手書きは対象マスの状態のみ定義、パイプラインは各パラメータ・方向分類・連続石数等を独立に分割している
- 最も差が小さいのは **DP（依存切替）** の4倍。手書き・パイプラインともに依存関係の観点は絞られやすい

## 3. 手書き→生成の1:1対応（全9件）

| # | 手書き (survey-design.md §5.3) | 生成 (test-requirements.yaml) | 対応強度 |
|---|---|---|---|
| 1 | BR-01 盤面の範囲外に対して False を返すこと | **BR-04-01** 範囲外の (x, y) で即座に False を返すこと | ◎ 完全一致 |
| 2 | BR-02 空でないマスに対して False を返すこと | **BR-04-02** 非空マスに対して False を返すこと | ◎ 完全一致 |
| 3 | BR-03 どの方向でも挟めない場合に False を返すこと | **BR-04-04** 空マスだが8方向すべて挟めない場合に False | ◎ 完全一致 |
| 4 | BR-04 少なくとも1方向で挟める場合に True を返すこと | **BR-04-03** 空マスかつ1方向以上で挟める場合に True | ◎ 完全一致 |
| 5 | BV-01 盤面の四隅（角）で正しく判定されること | **BV-04-01** 盤面の四隅（角）で正しく合法/非合法が判定されること | ◎ 完全一致 |
| 6 | BV-02 盤面の辺で正しく判定されること | **BV-04-02** 盤面の辺で正しく合法/非合法が判定されること | ◎ 完全一致 |
| 7 | EC-01 自分/相手/空 3種類の盤面状態を網羅 | **EC-04-01** 対象マスの3状態（空/自分/相手）を網羅 | ◎ 完全一致 |
| 8 | ER-01 空盤面で置こうとした場合に必ず False | **ER-04-02** 空盤面で全マスが False を返すこと | ◎ 完全一致 |
| 9 | DP-01 count_flippable が 0 の方向は is_valid_move の判定に寄与しない | **DP-04-03** count_flippable が 0 を返した方向は is_valid_move の判定に寄与しないこと | ◎ 完全一致 |

**対応精度**: 9/9 が意味的に一致。表現上の差異のみ（例: 「角」vs「四隅」）。

## 4. パイプラインが追加した36要求の分類

### 4.1 ヘルパー関数（is_on_board / is_empty / count_flippable）の独立要求

手書きは `is_valid_move` 単体を対象としたため、ヘルパー関数は対象外。パイプラインは選定4関数すべてを個別にテスト要求化している。

| 関数 | 追加要求数 | 内訳 |
|---|---|---|
| is_on_board (TGT-01) | 10 | BR×5, EC×1, BV×4 |
| is_empty (TGT-02) | 5 | BR×2, EC×2, ER×1 |
| count_flippable (TGT-03) | 15 | BR×9, EC×2, BV×2, ER×1, DP×1 |
| **小計** | **30** | |

### 4.2 is_valid_move に対するパイプラインの追加要求（6件）

| ID | 説明 | 手書きに無かった理由（推定） |
|---|---|---|
| BV-04-03 | x の範囲外直前直後（-1, 0, 7, 8）で判定が切り替わる | 四隅/辺の観点で既に包含と考えたため明示せず |
| EC-04-02 | 自分/相手（me=1, me=2）で対称動作 | 機能要件として暗黙視された |
| EC-04-03 | 挟める方向数 0 / 1+ の両条件 | BR-03/04 の観点で十分と判断した |
| ER-04-01 | 範囲外入力でクラッシュせず False | BR-01 に吸収 |
| ER-04-03 | 満盤で全マス False | 空盤面の対称ケースとして省略 |
| DP-04-01 | is_on_board 短絡（True時は is_empty スキップ等） | 実装詳細として省略 |
| DP-04-02 | is_empty の短絡 | 同上 |

## 5. パイプラインが手書きに欠落した要求

**該当なし（0件）**。手書きTRM 9件は全てパイプライン出力に含まれる。

## 6. 考察

### 6.1 カバー率の意味

パイプラインは人間が書く要求を **欠落させない**（リバーシサンプルにおいて）。これは本研究の問い (1)「十分性の向上」の一つの示唆となる。ただし単一サンプルでの結果のため、click/zod での追試が必要。

### 6.2 生成倍率 5× の解釈

パイプライン側 45件のうち、**ヘルパー関数に紐づく30件**（約67%）は、手書き TRM の対象範囲外である。これを除くと is_valid_move だけでの比較は **9 vs 15（1.7倍）** になる。

| スコープ | 手書き | パイプライン | 比率 |
|---|---|---|---|
| is_valid_move のみ | 9 | 15 | 1.7× |
| 全選定関数 | 9 | 45 | 5.0× |

本研究の問い (2)「非エンジニアへの情報共有」の観点では、is_valid_move に集中した **1.7倍の粒度差** の方が評価軸として意味を持つ。5倍は過大評価。

### 6.3 追加要求の品質

is_valid_move に対する6件の追加（§4.2）は、手書きでは **暗黙視されていた観点** を明示化している。具体的には:
- 対称性（自分/相手で動作が同じ）
- 短絡評価（ガード関数が False のとき次を呼ばない）
- 満盤ケース

これらはテスト設計のベテランなら手書きでも書くが、非経験者にとっては盲点になりやすい。**パイプラインの価値の一つは、経験差による抜け漏れを埋めること**と言える。

### 6.4 手書きTRMが持つ価値

一方、手書きTRMは9件に抑えられており、**非エンジニアに共有する際の可読性** で優位。パイプライン生成の45件を全部見せたら非エンジニアは読まない。

したがって実運用では:
- パイプラインで **45件を生成** → 十分性の確保
- そこから **Summary層（9件相当）** を抽出 → 非エンジニアに共有
- という **二段構成**（`complex-scope-approaches.md` の階層化パターン）が妥当

### 6.5 評価軸への反映

`trm-evaluation-criteria.md` の指標で本実験を表現すると:

| カテゴリA（十分性） | 値 |
|---|---|
| 手書き要求のカバー率 | 100% |
| 要求密度（件/100行） | 45 / 80 = 56.3件/100行 |
| 種別内訳 | BR20 / EC8 / BV9 / ER5 / DP4 |

| カテゴリB（可読性） | 値（推定） |
|---|---|
| Level 1（コード知識不要） | 手書き側 7/9 = 78%, パイプライン側 約20/45 = 44% |
| Level 2 | 手書き 2/9 = 22%, パイプライン 約15/45 = 33% |
| Level 3（コード知識必要） | 手書き 0/9 = 0%, パイプライン 約10/45 = 22% |

**可読率 (L1+L2)**: 手書き 100%, パイプライン 77%。
→ **パイプラインは詳細度を上げる代わりに可読性を落とす**傾向を明示。

## 7. 論文への示唆

### 7.1 主張しやすくなる論点

1. **パイプラインは人間が書くTRMを網羅する**（欠落なし）
2. **パイプラインは人間が暗黙視する観点を明示化する**（対称性・短絡評価等）
3. **粒度は人間の5倍程度に膨れる**が、対象を揃えれば1.7倍程度

### 7.2 注意すべき論点

1. **単一サンプルでの結果**。click/zod で再検証する必要あり
2. リバーシは合成題材で、実在コードの複雑度とは乖離がある
3. 手書きTRMは survey用に**意図的に9件に絞ってある**。真の比較には「人間が熟慮して書いた包括的TRM」との比較が必要

### 7.3 次稿論文での位置づけ

本実験の結果は、論文§6（実験結果）の **小節「パイプラインと手作業の比較」** として引用できる。
`paper-experiment-section.md` §4.2 に追記する候補段落:

> リバーシ合成サンプル（80行）に対して、熟慮した手書きTRM（9要求）とパイプライン自動生成TRM（45要求）を比較した。手書きの9要求は生成側の45要求に全て包含されており、カバー率は 100% であった。一方、パイプラインは対象関数以外のヘルパー関数（is_on_board, is_empty, count_flippable）に対しても要求を生成しており、約67%の要求はヘルパー関数由来の追加であった。同一関数（is_valid_move）への要求密度に限れば 9 vs 15 の 1.7 倍となり、パイプラインは熟慮された手作業よりわずかに包括的である。
